# Ecosystem Convergence

## What convergence creates

No single component in VisionClaw is novel. Solid pods exist. Git provenance exists. Nostr messaging exists. RDF ontology tooling exists. AI agent containers exist. Human-in-the-loop review exists. Knowledge graph visualisation exists.

What is novel is the closed loop that forms when these components share one identity primitive and one governance mechanism. The loop is:

**Human intent enters as natural language. An AI agent translates that intent into a conformant knowledge mutation. A human broker reviews the mutation as a visual diff. On approval, a git commit lands on the source pod with cryptographic provenance tracing the full chain -- who requested, who reasoned, who approved, which DID signed the push. The pod update propagates through a Nostr relay to subscribers. The knowledge graph re-indexes. The 3D physics view redraws. The ontology grows.**

No step in this loop requires the human to understand RDF, OWL, SPARQL, or git. No step bypasses human governance. Every step is identity-bound to a `did:nostr` keypair. Every step is auditable. The system compounds: each approved enrichment makes the next agent proposal better informed, and the Precedent mechanism progressively reduces broker load for routine enrichment types.

This document walks through the data lifecycle that makes this possible, maps each subsystem to its role, and explains how five independent repositories federate into one coherent platform.

## The data lifecycle

The lifecycle has seven phases. Each phase is owned by a distinct subsystem, but all share the `did:nostr:<64-hex>` identity primitive ([ADR-074](../adr/ADR-074-cross-system-did-nostr-canonicalisation.md)).

### Phase 1: Human intent

A knowledge base owner expresses intent through natural language -- voice (agentbox TTS/STT), text (chat, Nostr DM via nostr-rust-forum), or direct interaction with the Contributor Studio (`/studio` surface). The intent might be "add a relationship between smart contracts and formal verification" or "this note is mature enough to become a class".

The owner never edits triples, never writes SPARQL, never constructs URNs. The agent layer is the sole data manipulation surface ([PRD-013](../PRD-013-solid-git-ingest-surface.md) Section: Interaction Model).

### Phase 2: Agent reasoning

An AI agent in agentbox receives the intent. The agent holds its own `did:nostr` keypair (generated by `sovereign-bootstrap.py`, persisted to `/var/lib/agentbox/identities/`). It clones the knowledge base from the source pod via git-over-HTTP ([PRD-013](../PRD-013-solid-git-ingest-surface.md) G1), using NIP-98 auth headers derived from `did:nostr` credentials and signed by the server identity. The same NIP-98 credential flow secures outbound push operations during write-back.

The agent reasons over the knowledge base using the discovery engine ([PRD-009](../PRD-009-feature-engineering-discovery.md)): sentence embeddings for semantic similarity, N-hop edge materialisation for structural context, TransE vectors for gap detection. It produces a structured mutation -- a well-formed Turtle fragment, a valid `urn:visionclaw:` URI, a correct `owl:Class` hierarchy entry -- and validates it against schema constraints before submission.

If the mutation fails validation, it is rejected before any human sees it. The agent is the conformance layer.

### Phase 3: Broker case submission

The agent commits its proposed enrichment to a local branch and submits it as a `BrokerCase` with category `KnowledgeEnrichment` ([ADR-041](../adr/ADR-041-judgment-broker-workbench.md), extended by PRD-013 G4). The case carries:

- The enriched node's `urn:visionclaw:` URI
- The `did:nostr` of the proposing agent
- The from-state and to-state of the enrichment (e.g., `None -> Embedding`, `KGNode -> OntologyClass`)
- A reasoning summary and its SHA-256 hash

The Judgment Broker -- a distributed system spanning the DreamLab forum (governance.rs policy engine), agentbox (relay-consumer + broker-bridge endpoints), and VisionClaw (enrichment gating + BrokerActor on crashbug) -- adds the case to the broker's inbox. A WebSocket event (`broker:new_case`) pushes it to the review surface in real time. Approximately 65% of the broker is implemented across these three repos.

### Phase 4: Human review and decision

The broker sees the case in the Enrichment Review Pane -- a two-pane visual diff rendered in agentbox's linked objects viewer ([PRD-013](../PRD-013-solid-git-ingest-surface.md) G6). Source content on the left, proposed enrichment on the right. Markdown renders as markdown. Turtle highlights as Turtle. Embedding vectors display as tabular data. Below the diff, the provenance trailer block shows who proposed it, which agent reasoned over it, and the exact commit that will land.

Six decision outcomes are available ([ADR-041](../adr/ADR-041-judgment-broker-workbench.md)):

| Outcome | Effect |
|---------|--------|
| **Approve** | Write-back saga pushes the commit to the source pod |
| **Reject** | Case closed; no mutation |
| **Amend** | Broker modifies the enrichment; resubmits |
| **Delegate** | Routes to a domain expert for review |
| **Promote** | Approves and elevates -- e.g., a KGNode becomes an OWL class |
| **Precedent** | Approves and flags the enrichment pattern for future auto-approval |

The self-review invariant prevents the proposing agent's operator from approving their own enrichment. Every decision is recorded as an append-only `DecisionHistoryEntry`.

### Phase 5: Git commit with provenance

On Approve or Promote, the Write-Back Saga ([PRD-013](../PRD-013-solid-git-ingest-surface.md) G4) executes:

1. Fetch latest from the source remote to detect conflicts
2. Apply the enrichment to the local worktree
3. Commit with structured provenance trailers:

```
feat(ontology): promote vc:bc/smart-contract to OWL class

Urn: urn:visionclaw:concept:1a2b3c...:smart-contract
Proposed-by: did:nostr:4d5e6f...
Approved-by: did:nostr:7a8b9c...
Broker-case: case-2026-05-08-001
Reasoning-hash: sha256:abc123...
Signed-off-by: did:nostr:1a2b3c...
```

4. Push to the source pod via git-over-HTTP, with a NIP-98 signature (derived from `did:nostr` credentials) binding the transport to the pushing identity
5. Record the push result in the graph store (embedded Oxigraph) by creating a `WriteBackAudit` node linked via `WROTE_BACK_TO` relationship to the corresponding `GitRemote` node, capturing commit hash, saga status, and timestamp

The provenance trailers are parseable by `git log --format='%(trailers)'`. The commit history of the knowledge base becomes a complete audit trail of every machine-generated enrichment, traceable to specific agents, brokers, and DIDs.

### Phase 6: Relay notification

The broker's decision is emitted as three parallel representations:

| Representation | Durability | Audience |
|----------------|------------|----------|
| Git commit trailer | Permanent (repo history) | Anyone with repo access |
| Nostr bead (kind 30300) | Relay-durable | Relay subscribers across trust boundaries |
| Graph-store `DecisionHistoryEntry` (Oxigraph) | Application-durable | VisionClaw queries and KPI dashboards |

The Nostr event flows through the relay mesh ([ADR-073](../adr/ADR-073-private-nostr-relay-mesh-topology.md)): agentbox's embedded nostr-rs-relay, VisionClaw's ServerNostrActor, and optionally the nostr-rust-forum relay. Agents subscribed to approval events filtered by their `did:nostr` pubkey receive the notification. Human subscribers using any Nostr client can follow enrichment activity.

The IS-Envelope ([ADR-075](../adr/ADR-075-is-envelope-message-contract.md)) wraps all cross-system messages in a canonical JSON shape with seven kinds: `chat`, `tool_invoke`, `tool_result`, `knowledge_link`, `moderation`, `mesh_ping`, and `unknown`. NIP-59 gift-wrap encrypts the envelope on the wire when crossing relay boundaries.

### Phase 7: Graph re-index and visualisation

The knowledge graph re-indexes from the updated pod. The Insight Migration Loop ([insight-migration-loop.md](./insight-migration-loop.md)) detects the new state:

- If the enrichment was a Promote decision, the `BRIDGE_TO` edge flips from `candidate` to `promoted`. The promoted node begins settling toward the ontology core in the 3D physics view.
- If the enrichment added an embedding, the discovery engine can now find semantically similar concepts across domains.
- If the enrichment added a relationship, the GPU physics layout ([92 CUDA kernels](./physics-gpu-engine.md)) incorporates the new edge as a spring force.

The physics view is not decorative. Five semantic forces -- physicality clustering, role banding, maturity-driven z-axis, bridge-edge tension, and orphan drift -- make the migration state spatially readable. A broker can look at the 3D graph and see where enrichment effort should go next without running a query. Orphaned notes (no ontology anchor) drift into an amber zone. Recently promoted nodes visibly settle toward the authoritative core.

## The identity model

`did:nostr:<64-lowercase-hex>` is the universal identity primitive across every layer of the system ([ADR-074](../adr/ADR-074-cross-system-did-nostr-canonicalisation.md)). The hex value is the BIP-340 x-only Schnorr secp256k1 public key. Bech32 npub encoding is used only at the Nostr relay wire boundary; hex is canonical everywhere else.

Four classes of participant hold independent keypairs:

| Participant | Custody | Lifecycle |
|-------------|---------|-----------|
| **Pod owner** | Passkey-PRF-derived (forum) or extension-held | Long-lived; human-controlled |
| **VisionClaw server** | `SERVER_NOSTR_PRIVKEY` env var | Per-deployment; operator-controlled |
| **Agentbox agent** | Generated per container, persisted to `/var/lib/agentbox/identities/` | Per-agent; container-scoped |
| **Human broker** | Enterprise OIDC-bridged or Nostr NIP-07 extension | Per-session; human-controlled |

These four keypairs form the authority chain for every enrichment:

```
Pod Owner (did:nostr:<owner>)
    | WAC ACL grants read/write to:
    v
VisionClaw Server (did:nostr:<server>)
    | Signs NIP-98 on git clone/push; delegates to:
    v
Agentbox Agent (did:nostr:<agent>)
    | Proposes enrichments; commits carry Proposed-by trailer
    | Cannot push without approval from:
    v
Human Broker (did:nostr:<broker>)
    | Reviews via visual diff; decision carries Approved-by trailer
    v
Git Commit (signed push via server identity, trailers reference all four DIDs)
```

NIP-26 delegation ([ADR-074](../adr/ADR-074-cross-system-did-nostr-canonicalisation.md) Section D8) allows the server identity to delegate bounded authority to agent identities. The pod owner grants access to the server DID once; the server delegates to N agents without requiring the pod owner to ACL each one individually. Delegation tokens carry `kind` and `created_at` constraints so authority is time-bounded and scope-limited.

Every DID resolves to a Tier-3 DID Document at `/.well-known/did/nostr/<hex>.json`, advertising Solid pod, Nostr relay, and WebID service endpoints. A third party who knows only the hex pubkey can discover how to route messages and where to find the owner's data.

## Progressive trust: the Precedent path

The Judgment Broker's sixth decision outcome -- `Precedent` -- is the mechanism by which the system evolves from fully human-reviewed to selectively autonomous ([ADR-041](../adr/ADR-041-judgment-broker-workbench.md)).

When a broker approves an enrichment and marks it as Precedent, they are saying: "this pattern of enrichment is safe for future auto-approval." The system records the enrichment type, the proposing agent's class, the confidence score range, and the decision context. When a future enrichment matches the recorded pattern, the broker is not required to review it -- the system auto-approves based on the established precedent.

The progression is:

1. **Phase 0 -- Full review.** *(Complete.)* Every agent-proposed enrichment goes through the broker. This is the default and the only mode available at system startup.
2. **Phase 1 -- Pattern recognition.** *(Complete.)* After N approvals of the same enrichment type (e.g., embedding updates for well-understood domains), the broker marks the pattern as Precedent.
3. **Phase 2 -- Selective autonomy.** *(Complete.)* Routine enrichments (embedding updates, well-established ontology promotions) auto-approve. Novel enrichments (new relationship types, cross-domain promotions, structural changes) still require human review.
4. **Phase 3 -- Broker as exception handler.** *(Complete.)* The broker's inbox narrows to genuinely novel cases. The system handles routine knowledge maintenance autonomously within the bounds the broker has established.

This progression is strictly additive. Precedent can only be established by a human broker. It can be revoked at any time. The auto-approval path flows through the same audit trail as manual approval -- the commit provenance records `Decision: precedent-auto` instead of `Decision: approve`, so the audit trail distinguishes human decisions from precedent-matched ones.

The target metric ([PRD-013](../PRD-013-solid-git-ingest-surface.md)): 40% of routine enrichments auto-approved via Precedent by the end of the rollout.

## Cross-substrate federation

The five repositories in the DreamLab ecosystem are not a monolith. Each is an independent project with its own release cadence, its own Nix or Cargo build, and its own deployment topology. Federation is what makes them one system.

| Repository | Role | Relationship to VisionClaw |
|------------|------|---------------------------|
| **[VisionClaw](https://github.com/DreamLab-AI/VisionClaw)** | Integration substrate | This repo. Knowledge graph, GPU physics, broker workbench, discovery engine, write-back saga. |
| **[solid-pod-rs](https://github.com/DreamLab-AI/solid-pod-rs)** | Foundation library | Consumed as a Cargo dependency. Provides LDP, WAC, NIP-98 auth, git-http-backend, mashlib, and `did:nostr` resolution. |
| **[agentbox](https://github.com/DreamLab-AI/agentbox)** | Agent container | Git submodule. Provides sovereign agent execution, nostr-rs-relay, linked objects viewer, adapter-based integration. Ontology bridge (10 MCP tools) proxies agents to VisionClaw's Oxigraph SPARQL store. Browser-based setup wizard (PRD-012). |
| **[nostr-rust-forum](https://github.com/DreamLab-AI/nostr-rust-forum)** | Forum kit | 11 `nostr-bbs-*` Rust crates. Provides human-to-human and human-to-agent text coordination, Nostr relay, community moderation. |
| **[dreamlab-ai-website](https://github.com/DreamLab-AI/dreamlab-ai-website)** | Branded deployment | Downstream consumer of the forum kit. React SPA + WASM forum + `forum-config/` package. |

Three binding contracts hold the federation together:

1. **`did:nostr:<hex>` identity** ([ADR-074](../adr/ADR-074-cross-system-did-nostr-canonicalisation.md)). Every substrate emits the same DID Document shape with `SchnorrSecp256k1VerificationKey2019`. Every WAC ACL, every Nostr event signature, every git push auth header uses the same 64-char lowercase hex pubkey.

2. **IS-Envelope v1** ([ADR-075](../adr/ADR-075-is-envelope-message-contract.md)). Every cross-system message uses the same JSON shape with seven kinds. JCS-canonicalised for deterministic hashing. NIP-59 gift-wrapped on the wire. Mapped losslessly to ActivityStreams 2.0 LDN at the Solid pod inbox boundary.

3. **Private relay mesh** ([ADR-073](../adr/ADR-073-private-nostr-relay-mesh-topology.md)). NIP-42 AUTH-gated relays in each substrate, federated via peer relay lists. Three deployment modes (standalone, federated, client) configured per-substrate without recompilation.

Two parallel URI namespaces coexist by design:

- `urn:visionclaw:<kind>:<hex-pubkey>:<local>` -- 6 kinds, minted in `src/uri/`
- `urn:agentbox:<kind>:[<hex-pubkey>:]<local>` -- 18 kinds, minted in `management-api/lib/uris.js`

The BC20 anti-corruption layer ([DDD context map](../ddd-mesh-federation-context.md)) translates between namespaces at the federation boundary. Both share `did:nostr:<hex>` as the identity primitive and `sha256-12-<12 hex chars>` for content addressing.
